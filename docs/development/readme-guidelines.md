# README 维护准则

这份文档记录 CodexPilot README 的信息架构和文案判断。README 是项目首页，不是完整手册；功能细节应优先维护在 `docs/features.md` 和 `docs/features.en.md`。

## 目标

README 第一屏要快速回答三个问题：

1. CodexPilot 是什么。
2. 用户怎么开始使用。
3. 它为什么值得信任。

不要把 README 写成完整功能清单、发布记录或开发手册。详细说明可以通过文档入口承接。

## 首屏结构

README 顶部保持产品化但克制的结构：

1. 居中图标。
2. 居中项目名。
3. 一句话定位。
4. 语言切换。
5. 徽章。
6. 一段补充说明。
7. 官方关系和安全边界提示。
8. 一张管理器总览图。

图标使用仓库内已有应用图标，避免额外引入只服务 README 的装饰资产。首屏只保留一张总览截图；如果当前版本需要重点解释某个核心能力，可以在对应小节补一到两张真实页面截图，但不要把 README 重新写成完整截图手册。

## 信息顺序

当前 README 顺序为：

1. 快速使用。
2. 核心亮点。
3. 其他功能。
4. 本地数据与安全。
5. 文档入口。
6. 交流与支持。
7. License。

快速使用应放在前面。用户第一次打开 README 时，优先需要知道安装后如何启动和进入关键页面；完整安装细节和功能细节可以稍后阅读。

快速使用要面向普通用户：

- 快速使用同时承接下载和首次打开，不再单独保留“安装”章节。
- 明确让用户去 GitHub Releases 的 Assets 区下载平台安装包，不要下载 Source code 压缩包。
- Windows/macOS Apple Silicon 的安装步骤写清楚，macOS 未签名、公证和 Intel 限制只做必要提醒。
- 源码运行、测试命令和 UI 预览不放在公开 README；这些属于维护者工作流，应放在内部开发文档或本地记录里。

功能说明文档应保留必要截图。README 默认只放管理器总览图；如果核心卖点需要降低理解成本，可以在 README 的重点能力小节放关键截图。`docs/features.md` 和 `docs/features.en.md` 可以按功能页放启动、模型通道、对话维护和诊断截图，帮助读者把文字说明和真实界面对上。

## 文案定位

CodexPilot 的定位应突出本地控制和可解释操作：

- 本地启动。
- 对话维护。
- 模型通道管理。
- Provider 归属同步。
- 诊断。
- 不修改 Codex App 安装目录。

避免使用过泛的“外部增强”“让 Codex 更强/更好用”等表达。这类表达容易和其他 Codex 增强项目混在一起，也弱化了 CodexPilot 对本地数据、安全边界和手动控制的强调。

推荐句式：

> 让 Codex 的本地工作流更顺手、更可控。

补充说明可以写：

> CodexPilot 提供一个本地管理界面，用 Chromium DevTools Protocol 连接正在运行的 Codex 页面。你可以从这里启动 Codex、导出会话、处理回收站、同步 Provider 归属、配置混合中转和查看诊断日志；它不修改 Codex App 安装目录，也不替换 Codex 本身。

## README 与功能文档分工

README 保留首页级内容：

- 项目定位。
- 快速使用。
- 核心亮点。当前重点是混合中转和 Provider 归属同步，要比普通功能讲得更具体。
- 其他功能只保留名称列表，避免抢核心亮点的阅读重心。
- 本地数据与安全摘要。
- 文档链接。
- 支持入口。

`docs/features.md` 保留完整功能说明：

- 启动与注入。
- 会话导出与维护。
- 模型通道。
- Provider 归属同步。
- 诊断。
- 本地数据与安全。
- 兼容性说明。

新增功能时，先判断它属于首页级卖点还是说明书细节。大多数新增功能应先写入功能说明文档；只有当它影响用户首次理解项目时，才同步进入 README 的核心功能列表。

## 中英文同步

中文 README 和英文 README 应保持同一套结构。新增或移动章节时，需要同步检查：

- `README.md`
- `README.en.md`
- `docs/features.md`
- `docs/features.en.md`
- README 中链接到的公开文档

英文 README 不应链接到中文功能说明作为唯一入口。双语读者路径要完整闭环。

## 不建议放进 README 的内容

- 过长的功能教程。
- 多张页面级截图。
- 详细配置文件说明。
- 发布前检查清单。
- 打包和发布内部流程。
- 兼容性细节和风险展开。
- 临时工作计划。

这些内容应分别进入功能说明、开发文档或发布文档。

## 更新检查

每次改 README 后至少检查：

- 顶部图标路径是否存在。
- 总览截图路径是否存在。
- 中英文 README 结构是否一致。
- README 是否仍短于完整手册。
- 功能说明文档是否承接了被移出的细节。
- 文档链接和标题锚点是否可读。
- 是否出现过于接近其他项目的定位话术。
